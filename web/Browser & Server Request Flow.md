## 배운 것
브라우저와 서버가 요청/응답을 주고받는 구조

## 왜 중요한가?
프론트(HTML/CSS/JS)와 백엔드(Java + Spring)가 각각 어떤 역할을 나누어 맡고, 하나의 요청이 어떤 계층을 거쳐 처리되는지 이해하면 전체 웹 동작을 그림으로 그릴 수 있다.

## 새롭게 알게 된 정보
1. JavaScript는 브라우저에서 동작하는 언어로, 자바와는 다른 언어다. 싱글 쓰레드로 동작하지만 브라우저의 이벤트 루프를 활용해 화면이 멈추는 현상을 막는다.
2. 브라우저는 렌더링 엔진, JS 엔진, 네트워크 스택, DOM(HTML/CSS와 JS를 잇는 다리), 저장소, 보안 샌드박스를 한 몸에 갖춘 실행 환경이다.
3. C++이나 Java에서는 GUI(Qt, Swing/AWT 등)를 별도 라이브러리로 붙여야 하지만, 브라우저는 렌더링·GUI·네트워킹·샌드박싱이 기본으로 통합되어 있다.
4. 서버(Spring)는 요청을 Controller → Service → Repository 순서로 처리하고, 그 결과를 다시 Controller가 JSON으로 직렬화해 응답한다.

## 예시

**브라우저 구조**
```
┌─────────── 브라우저 ───────────┐
│                               │
│  HTML/CSS  ←── DOM ──→ JS     │
│  (화면 구조/스타일)   (동작 로직) │
│                        │      │
│                   axios로 요청 │
└───────────────────┼───────────┘
                    │ 요청 ↓
                    │ 응답 ↑
                    ▼
              ┌───────────┐
              │    서버    │
              └───────────┘
```

**서버 요청 처리 흐름 (Java + Spring)**
```
┌────────────────── 서버 (Java + Spring) ──────────────────┐
│                                                            │
│  요청 도착 (POST /api/auth/login, body: {id, password})    │
│         ↓                                                  │
│  ① Controller (@RestController, @PostMapping)              │
│     → URL 라우팅, 요청 바디를 Java 객체로 자동 변환           │
│         ↓                                                  │
│  ② Service (비즈니스 로직)                                  │
│     → 비밀번호 검증, 토큰 생성 등 실제 "로그인" 로직 처리       │
│         ↓                                                  │
│  ③ Repository / DAO (JPA, MyBatis 등)                      │
│     → DB에서 사용자 정보 조회                                │
│         ↓                                                  │
│  ④ Controller가 결과를 JSON으로 직렬화해서 응답               │
│     { "success": true, "data": {...} }                     │
│                                                            │
└────────────────────────────────────────────────────────────┘
```
